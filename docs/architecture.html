Arquitectura de Software: RPi Door Access RFID

1. Visión General

El sistema es una aplicación IoT desplegada en una Raspberry Pi para gestionar el acceso a una puerta mediante tecnología RFID. El software está diseñado bajo los principios SOLID y conceptos de Clean Architecture, asegurando un sistema robusto, modular y altamente testeable sin necesidad de tener el hardware físico acoplado durante el desarrollo.

2. Stack Tecnológico

  • Lenguaje básico: Python 3
  • Framework Web y API: FastAPI
  • Base de Datos: SQLite (local)

3. Capas de la Arquitectura

El código está organizado de manera limpia en múltiples capas (reflejadas en la estructura de la carpeta app/):

  • Domain (Dominio): Contiene las entidades y las reglas de negocio puras. Esta capa es independiente del hardware, el framework web o de la base de datos (por ejemplo, lógica de validación de créditos).
  • Infrastructure (Infraestructura): Implementaciones concretas de acceso a los datos (SQLite) y controladores de hardware (GPIO para LED, Buzzer, Relé, Lector RFID).
  • Core: Módulos de configuración, inyección de dependencias y utilidades compartidas siguiendo el principio DRY.
  • API: Endpoints y controladores de FastAPI que exponen servicios REST y sirven la interfaz web local.

4. Abstracción de Hardware y Principios SOLID

Siguiendo estrictamente el principio Open/Closed, toda la lógica que interactúa con el hardware físico ha sido abstraída a través de interfaces o clases abstractas:

  • LedController
  • BuzzerController
  • RelayController
  • RFIDScanner

Estas abstracciones se inyectan en las clases de servicio. Esto aísla el comportamiento técnico de la Pi de la lógica de evaluación de accesos y la manipulación de base de datos, facilitando migraciones en un futuro (ej. de GPIO clásico a I2C).

5. Gestión de Datos

El almacenamiento se gestiona localmente de forma desconectada mediante SQLite, definiendo tres casos de uso o tablas clave:

  1. Validación (Private DB): Almacena IDs del RFID, estado de la cuenta (activo/inactivo), fechas de expiración y saldo actual de créditos.
  2. Logs Cíclicos: Registro de todo intento de lectura de tarjeta (concedida o denegada) para auditoría.
  3. Tracking de Límites: Registro histórico particularizado de la métrica y recuentos de uso o límites diarios.

6. Escalabilidad Futura (AWS)

Gracias al diseño desacoplado de las interfaces, la arquitectura en un futuro escalará y sincronizará vía servicio telemétrico (AWS IoT Core). El sistema continuará trabajando de manera local mediante caché si se pierde la conexión, ya que la lógica interna evalúa la validación con la base de datos SQLite embebida de manera primaria.
